# 중앙 오케스트레이터 패턴의 한계

> **왜 하나의 중앙 에이전트가 여러 전문 에이전트를 관리하기 어려운가?**

---

## 핵심 문제점 5가지

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│     ┌───────────┐                                               │
│     │ 전문      │                                               │
│     │ 에이전트1 │◄──┐                                           │
│     └───────────┘   │                                           │
│                     │      ┌──────────────────┐                 │
│     ┌───────────┐   │      │                  │                 │
│     │ 전문      │◄──┼──────│  중앙            │  ← 병목 지점    │
│     │ 에이전트2 │   │      │  오케스트레이터  │  ← 단일장애점   │
│     └───────────┘   │      │                  │                 │
│                     │      └──────────────────┘                 │
│     ┌───────────┐   │                                           │
│     │ 전문      │◄──┘                                           │
│     │ 에이전트N │                                               │
│     └───────────┘                                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

| 문제 | 설명 |
|:----:|------|
| **1. 컨텍스트 병목** | 모든 에이전트 출력을 이해·종합해야 하나, 컨텍스트 윈도우 제한으로 한계 |
| **2. 작업 분배 복잡성** | 어떤 작업을 누구에게? 의존성 파악과 순서 결정이 어려움 |
| **3. 상태 관리** | 여러 에이전트의 진행 상태 동시 추적 + 일관성 유지 어려움 |
| **4. 통신 오버헤드** | 에이전트 수 증가 → 조정 비용 **기하급수적 증가** |
| **5. 단일 장애점** | 오케스트레이터 실패 = 전체 시스템 중단 |

---

## 대안적 접근법

| 패턴 | 특징 |
|:----:|------|
| **계층적 오케스트레이션** | 중간 관리자 에이전트를 두어 부담 분산 |
| **P2P 협업** | 에이전트 간 직접 통신, 중앙 의존도 감소 |
| **이벤트 기반** | 메시지 큐를 통한 느슨한 결합 (Loose Coupling) |

---

> [!tip] 핵심 메시지
> **"만능 관리자"보다 "역할 분담"이 확장성과 안정성 모두에서 유리하다**
